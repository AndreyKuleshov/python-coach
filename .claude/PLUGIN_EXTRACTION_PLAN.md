# Plan: split the build harness from the application

Status: **planning ‚Äî execute when the harness repo exists.** The owner will create a
separate harness repo; this plan guides the extraction and the consumption model.
The revisit trigger from the original plan is met (the agents have driven many
features end-to-end with stable prompts). Updated 2026-06-17.

## Vision

Separate the project into two things:

- **The harness (engine)** ‚Äî the reusable *factory*: the agent suite, the coding
  rules, the QA references, the platform blueprint, and the devops agent. It lives
  in its **own repo** and is consumed as a **dev-only dependency**. It is never part
  of any deployed application.
- **The application** ‚Äî a concrete learning platform. Its only unique inputs are
  **`sources/`** (the subject's books) and **`CONTEXT.md`** (the subject, learner
  profile, target skill). Everything else ‚Äî schema, API, sandbox, frontend, auth,
  progression, profile, i18n, theme, AI features, lessons ‚Äî is **generated by the
  harness agents** from those two inputs, guided by the blueprint.

> The whole point: to build a new learning platform for any subject, you drop new
> books into `sources/`, write `CONTEXT.md`, install the harness, and let the agents
> build it ‚Äî inheriting the full feature set automatically. At deploy time the
> harness is gone; only the platform ships.

## Confirmed Claude Code facts (from claude-code-guide, 2026-06-17)

These shape the design ‚Äî do not plan against features that don't exist:

- A **plugin can bundle**: `agents/`, `skills/`, `commands/`, `hooks/hooks.json`,
  `.mcp.json`, `.lsp.json`, `bin/`, `settings.json`, manifest in `.claude-plugin/`.
- A **plugin CANNOT ship path-globbed `.claude/rules/*.md`** (auto-applied rules).
  Rules load only from **project-level `.claude/rules/`** and **user-level
  `~/.claude/rules/`** (and `paths:` in user-level rules has a known bug). So shared
  rules must be **delivered into the project's `.claude/rules/`**, not via the plugin.
- **Precedence:** project `.claude/agents/` overrides same-named **plugin** agents,
  which override user-level. So the harness ships baseline agents; an app repo can
  override any of them locally.
- **Plugins live in the user's Claude config**, not in the app repo's working tree ‚Äî
  so they are **inherently absent from a deployed artifact** (a Docker image of the
  service). Local `.claude/` in the repo is also dev-only (never executed at runtime),
  but we still keep it out of the build context.
- No `${PLUGIN_ROOT}` for agents (only skills get `${CLAUDE_SKILL_DIR}`) ‚Äî so
  agent-read reference files must be repackaged as a **skill** inside the plugin.

## What moves to the harness vs stays in the app

| Harness repo (the dependency) | Application repo (built + deployed) |
|---|---|
| `agents/`: `python-platform-architect`, `methodist`, `coder`, `code-reviewer`, `qa`, **`devops`** | `sources/` (subject books) |
| The coding **rules** (`python`, `all-languages`, `api-layers`, `stack`, `pyproject`, `dotenv`, `docker`, `makefile`) ‚Äî shipped as harness files + a sync mechanism (see below), NOT a plugin rules dir | `CONTEXT.md` (subject/goal/skill ‚Äî the planner's input) |
| QA references (`flakiness`, `selectors`, ‚Ä¶) repackaged as a **plugin skill** (so agents can read them via `${CLAUDE_SKILL_DIR}`) | The generated platform: `services/`, `deploy/`, `dev/`, `Makefile`, `pyproject.toml`, `README.md`, `STORAGE_CONTRACT.md`, `CURRICULUM.md`, `fixtures/` |
| `docs/PLATFORM_BLUEPRINT.md` ‚Äî the architect's end-to-end planning reference + feature catalog | `docs/bugs/` (this platform's own defect registry) |
| A scaffold/bootstrap (seed a new app repo: empty `sources/`, `CONTEXT.md` template, synced rules) | A thin `.claude/` holding only project-specific overrides (usually none) + enabling the plugin |

## Distribution & consumption model

- **Agents + QA-refs-skill + hooks ‚Üí a Claude Code plugin** in the harness repo,
  distributed via a git **marketplace**; an app repo installs it with `/plugin install`.
- **Rules ‚Üí delivered into the app's `.claude/rules/`** (plugins can't auto-apply
  rules). Two viable mechanisms ‚Äî pick one at extraction time and validate it:
  1. a **plugin SessionStart hook** (or a `bin/` command like `harness-sync-rules`)
     that copies the harness's bundled rule files into the project's `.claude/rules/`;
  2. a **scaffold step** when creating the app repo that copies the rules in (and a
     re-sync command to update them). Mechanism (1) keeps a single source of truth;
     (2) is simpler but drifts.
- **Blueprint** ships in the harness; the **architect reads it** (now wired into the
  architect prompt) to plan the end-to-end build from `CONTEXT.md` + `sources/`.

## The agent pipeline (harness-driven build)

architect (reads `CONTEXT.md` + `sources/` + blueprint ‚Üí end-to-end plan + builds
infra/contract) ‚Üí methodist (curriculum + bilingual lessons from `sources/`) ‚Üí
coder (features) ‚Üí qa (tests) ‚Üí code-reviewer (gates/security) ‚Üí **devops**
(package + deploy the app only). The architect is the planner; the rest execute.

## Deploy excludes the harness (requirement)

- The plugin is dev tooling in Claude config ‚Üí never in the deploy artifact. ‚úì
- The **devops agent** (now in `.claude/agents/devops.md`) owns the guarantee: a
  `.dockerignore` / multi-stage build excludes `.claude/`, `sources/`, `docs/`,
  `tests/`, `.git`, `.env*` from the image; the deploy artifact is only the
  `services/api` service + the `services/sandbox` image + the database. It verifies
  the image contains no harness/build files and no baked secrets.
- The harness repo is separate, so the app repo can even be cloned without it for a
  prod build.

## Open design risks to validate at extraction time

1. **Rules delivery** (the main one): plugins can't ship auto-applying rules ‚Äî build
   and test the sync mechanism (hook or scaffold) so generic agents still get the
   path-globbed rules in every consuming repo.
2. **QA references**: must become a plugin **skill** (no `${PLUGIN_ROOT}` for agents);
   update the `qa`/`code-reviewer` prompts to read them via the skill, not
   `.claude/references/qa/`.
3. **Subject-agnostic prompts**: the architect/methodist prompts still carry
   Python/AQA specifics ‚Äî generalize them to read the subject from `CONTEXT.md`
   (the architect already treats `CONTEXT.md` as authority; finish the job for both).
4. **Plugin frontmatter limits**: no `hooks`/`mcpServers`/`permissionMode` in agent
   frontmatter; the deferred pre-commit hook stays a project/plugin-hook concern.

## Extraction steps (when the harness repo is ready)

1. Create the harness repo with `.claude-plugin/plugin.json`, `agents/` (the six
   agents), `skills/` (QA refs + any), `hooks/` (rules-sync), `docs/PLATFORM_BLUEPRINT.md`,
   the canonical `rules/`, and a scaffold/bootstrap.
2. Stand up a git **marketplace** for it.
3. In a consuming app repo: `/plugin install`, run the rules-sync, drop `sources/` +
   write `CONTEXT.md`, then run the architect to plan + build.
4. Migrate THIS repo to the model: remove the harness agents/rules/refs/blueprint
   from `.claude/` once the plugin + rules-sync provide them; keep only `sources/`,
   `CONTEXT.md`, the built app, and `docs/bugs/`.
5. Wire devops `.dockerignore`/build so the deploy artifact excludes everything harness.
